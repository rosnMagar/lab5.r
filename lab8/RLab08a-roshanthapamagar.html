Home Prices

The response variable in our analysis is price. Let’s graph it.

raw.ames %>% ggplot() + geom_histogram(mapping = aes(x=Sale_Price))

Note that there are a few really expensive houses. If we don’t predict these houses very well, they might make our error estimates huge. Or, put another way, if we tune our predictive models to minimize error, we may end up overfitting these outliers at the expense of other data points. One way to avoid this is to transform our price variable using the logarithm function.

Side-note on Logarithms

Remember that a logarithm function \(\log_a\) is defined as follows: \(\log_a(x) = b\) if and only if \(a^b = x\). [Note that these formulas will look right in the knit version of this document.] In other words, \(\log_a(x)\) is the power you’d have to raise \(a\) to in order to get \(x\). This is read as “log base a.”

There are two bases normally used with logarithms: “base 10” and “base e.” Remember that e is the special number with non-terminating decimal expansion e=2.718282…. Logarithms base e have a special place in Calculus, have nice theoretical properties, and are called “natural logarithms.” In calculus, they’re represented as the ln function, but in R, the log function represents natural logarithms.

Important features of the natural log function include:

  • log(x) is only defined for positive values of x.
  • Logarithms are strictly increasing as x increases.
  • A log transform will rescale a variable, often changing right-skewed data into a more symmetric form.
  • Because the transformation can be undone, if we can predict log(x) for some variable, we can do the inverse transform at the end to predict x: \(e^{\log(x)} = x\).

In R code, the exp function is the inverse of the log function. Let’s try it:

log(372)
[1] 5.918894
exp(log(372))
[1] 372

Question 1

Based on the table, do the two Overall variables seem to go together? Briefly explain why this might be helpful or not helpful.

Looking at the table, the two Overall variables (Overall_Qual and Overall_Cond) are related but not perfectly matched. Most houses cluster around Average condition with Average quality (300 houses in that cell), which shows they often go together. However, there are many houses where they don’t match - for example, 456 houses have Average condition but Good quality, and 178 houses have Good condition with Good quality. This is helpful because they measure different things: Overall_Qual tells us about the building materials and construction quality, while Overall_Cond tells us about how well the house has been maintained. A house can be well-built (high quality) but poorly maintained (low condition), or vice versa. Having both variables helps the model understand that a well-built but run-down house might have different value than a moderately-built but well-maintained house.

A Random Forest to Predict Price

There are two schools of thought concerning inclusion of variables in a random forest. One school of thought suggests it’s good to remove variables from the model if those variables are (a) not related very strongly to the response variable, or (b) highly correlated with other predictor variables. This advice is definitely good if you’re dealing with regression models.

On the other hand, another school of thought suggests that random forests are actually very good at handling extraneous variables and highly correlated predictor variables. My experience has been that a large enough and appropriately tuned random forest model gets better predictive accuracy from including as many variables as possible. The trade-off in this approach is that you need to tune model parameters and spend computation time on a data set with more variables.

For now, let’s compute a random forest predictor using all our variables and default settings. The randomForest command can be run using a similar format to our other predictor commands:

randomForest(response ~ predictors, data=data.frame)

Three Measures of Error

How well does the model do at predicting log.price? Let’s measure it three ways.

The Wrong Way

We could measure the RMSE of the random forest applied to its original training set. This is the wrong way to do it. But, let’s try it anyway. Note that the Metrics package gives us the rmse command to easily compare actual log.price to predicted.

 train.predict <- predict(ames.rf, newdata=ames.train)
 rmse(ames.train$log.price, train.predict)
[1] 0.05446083

Out-of-Bag Error

Honest measures of prediction error will never measure a prediction model against data that was used to create the model. Using cross-validation or out-of-bag error are both honest ways to use the training set to get a measure of error, but out-of-bag error is the easiest with random forests. You can get out-of-bag error from the default model summary or by printing the last entry of the mse field of the model directly. It’s not labeled as “out-of-bag error” but it is. Since the randomForest procedure gives mean squared error, rather than RMSE, we’ll take the square root of the number we got from the randomForest output.

Measuring Error with a Test Set

Another honest way to measure prediction error is to apply the model to a separate testing set that wasn’t used to create the model. The code below loads in our testing set, applies the same modifications to it, and measures how well our model does with this new data.

test.predictCART <- predict(ames.tree, newdata=ames.test)
rmse(ames.test$log.price, test.predictCART)
[1] 0.2195959
test.predict <- predict(ames.rf, newdata=ames.test)
rmse(ames.test$log.price, test.predict)
[1] 0.1406617

You can see that the Random Forest totally beat the CART Tree with the test data. Using the Test set gave a similar answer to the OOB, which is handy, since OOB is “free” in a rf model.

Question 4

Compare these three measures of error. Which match, and which don’t? Which are higher and which are lower? Why is that important?

The three error measures show different values. The training RMSE (wrong way) is 0.055, which is the lowest. This is because the model was built using this same data, so it fits it very well i.e overfits. The out-of-bag (OOB) RMSE is 0.136, which is higher. This is an honest estimate because each tree in the random forest only uses part of the training data, so it can test itself on data it hasn’t seen. The test set RMSE is 0.141, which is very similar to the OOB RMSE (0.136 vs 0.141).

The OOB and test set RMSE match closely, which is important because it shows that OOB error is a reliable way to estimate how well the model will do on new data, without needing to set aside a separate test set. The training RMSE being much lower (0.055) than the other two (around 0.14) indicates overfitting - the model memorized the training data too well and won’t perform as well on new houses. This is why we need honest error measures like OOB or test set RMSE to know how good our model really is.
